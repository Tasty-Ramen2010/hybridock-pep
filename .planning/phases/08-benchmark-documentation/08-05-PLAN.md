OPEN QUESTION 1: MDAnalysis GPL-2.0 in rapidock-env
----------------------------------------------------
MDAnalysis is GPL-2.0 licensed. GPL is OSI-approved but has strong copyleft.
HybriDock-Pep's own source (score-env) does NOT import MDAnalysis directly.
MDAnalysis lives in rapidock-env, used by RAPiDock's internal trajectory analysis.
The driver calls RAPiDock via `conda run -n rapidock-env python run_rapidock.py`
(a subprocess boundary — GPL does not propagate across subprocess boundaries under
the GPL's own FAQ: "pipes, sockets and command-line arguments" create no obligation).

However: `src/hybridock_pep/sampling/run_rapidock.py` is OUR source code running
INSIDE rapidock-env where MDAnalysis is present. If RAPiDock transitively imports
MDAnalysis when run_rapidock.py runs, iGEM's rubric may interpret this as a copyleft obligation.

RAM'S RULING NEEDED:
  Option A: Accept — GPL in rapidock-env subprocess is acceptable under iGEM's "OSI-approved"
            interpretation. Document rationale above in final licenses.txt.
  Option B: Remove — replace MDAnalysis usage in rapidock-env with a permissive alternative
            (requires RAPiDock fork or MDAnalysis-free path). High implementation cost.
  Option C: Clarify — ask iGEM judges directly before submission.
